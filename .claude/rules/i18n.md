# Internationalization (i18n)

This project supports English (en-US, default) and Spanish (es) locales using Angular's built-in `@angular/localize` system.

## Locale Setup

### Supported Locales

- **en-US** (default) — English locale ID; shipped by default
- **es** (Spanish) — Spanish locale ID; separate build artifact

Stored in `apps/my-personal-project/src/locales/`:
- `messages.json` — English strings (default)
- `messages.es.json` — Spanish translations

## Marking Strings for Translation

### Template Strings

Use `$localize` with backticks in component templates (Angular syntax):

```html
<!-- Good: Angular $localize syntax -->
<p i18n>Welcome to the app</p>
<h1 i18n="@@page.title">Article</h1>
<button i18n-title="button action" title="Save article">Save</button>
```

Or in TypeScript:

```typescript
import { $localize } from '@angular/localize/init';

const message = $localize`Hello, ${name}!`;
const greeting = $localize`:@@greeting|Greeting message:Hello!`;
```

### Message IDs

Provide explicit IDs for stability across changes:

```html
<p i18n="@@article.title">My Article</p>
```

This creates message ID `article.title` in the xliff/json extraction.

### Parameterization

For dynamic values:

```html
<!-- Interpolation -->
<p i18n>Welcome, {{ userName }}!</p>

<!-- ICU message format (plural, select) -->
<p i18n>
  {itemCount, plural, one {1 item} other {# items}}
</p>
```

## Build and Translation Workflow

### Extract Strings

```bash
# Extract all i18n strings into xliff files (not typically done in this project)
ng xi18n
```

In this project, extraction is implicit during build.

### Build Localized Versions

```bash
npm run localize
```

This command:
1. Runs `nx build --localize`
2. Generates separate build artifacts for each locale:
   - `dist/apps/my-personal-project/browser/` (default en-US)
   - `dist/apps/my-personal-project/browser/es/` (Spanish)
3. Serves the localized app at `http://localhost:4200` (by default)

### Message Files

Translation keys are stored in `messages.*.json` files:

**`messages.json` (English, default)**:
```json
{
  "article.title": "My Article",
  "article.description": "A great article",
  "button.save": "Save",
  "greeting": "Hello, {name}!"
}
```

**`messages.es.json` (Spanish)**:
```json
{
  "article.title": "Mi Artículo",
  "article.description": "Un excelente artículo",
  "button.save": "Guardar",
  "greeting": "¡Hola, {name}!"
}
```

### Adding Translations

1. Mark new strings with `i18n="@@unique.id"` in templates or `$localize` in TypeScript
2. Add translation entry to `messages.es.json`:
   ```json
   {
     "your.new.key": "Your translated text"
   }
```
3. Run `npm run localize` to rebuild and verify

## Runtime Locale Switching

Locale is determined at build time (separate artifacts per locale). To switch locales at runtime:

### Using `LOCALE_ID`

In components, inject `LOCALE_ID` to read the current locale:

```typescript
import { LOCALE_ID } from '@angular/core';

export class LanguageSwitcherComponent {
  readonly currentLocale = signal(inject(LOCALE_ID));
}
```

### Switching Locales

To switch, redirect the browser to the appropriate locale path:

```typescript
export class LanguageSwitcherComponent {
  switchToSpanish() {
    // Redirect to Spanish build artifact
    window.location.href = '/es/';
  }

  switchToEnglish() {
    // Redirect to English (root)
    window.location.href = '/';
  }
}
```

## Firebase Hosting Rewrites

The `firebase.json` file configures per-locale routing:

```json
{
  "hosting": {
    "rewrites": [
      {
        "source": "/es/**",
        "destination": "/es/index.html"
      },
      {
        "source": "/**",
        "destination": "/en-US/index.html"
      }
    ]
  }
}
```

**How it works**:
- Requests to `/es/*` → serve Spanish index.html and assets from `dist/.../.../es/`
- All other requests → serve English index.html and assets from `dist/.../browser/`
- Each locale is a self-contained single-page app

## Deployment Workflow

1. **Local Development**:
   ```bash
   npm run localize  # Builds both en-US and es versions
   # Serves at http://localhost:4200 (en-US by default)
   # Access /es/ for Spanish
   ```

2. **Production Deployment**:
   ```bash
   npm run deploy  # Builds and deploys to Firebase Hosting
   ```
   Firebase distributes the localized bundles per the rewrites rules above.

3. **Verify Locales**:
   ```bash
   # Visit https://my-personal-project.firebaseapp.com/
   # Visit https://my-personal-project.firebaseapp.com/es/
   ```

## Best Practices

### Keep Translations in Sync

Maintain `messages.json` and `messages.es.json` in parallel. If a key exists in English but not Spanish, add a TODO or track it:

```json
{
  "new.feature": "New Feature",
  "new.feature.es": "TODO: Translate"
}
```

### Use Descriptive Message IDs

```html
<!-- Good -->
<p i18n="@@article.empty_state">No articles found</p>

<!-- Avoid -->
<p i18n>No articles found</p>
```

Explicit IDs make extraction, translation, and future refactoring easier.

### Pluralization

For plurals, use ICU message format:

```html
<p i18n>
  {count, plural, =0 {No items} one {1 item} other {# items}}
</p>
```

### Dynamic Values

Interpolate safely:

```html
<p i18n>Welcome back, {{ firstName }}!</p>
```

Angular localizer handles variable substitution automatically.

### Testing i18n

In tests, locale is determined by the test harness. To test a specific locale:

```typescript
import { LOCALE_ID } from '@angular/core';

TestBed.configureTestingModule({
  providers: [
    { provide: LOCALE_ID, useValue: 'es' }
  ]
});
```

## Locale Data and Formatting

For date/number formatting specific to locale:

```typescript
import { formatDate, formatNumber } from '@angular/common';

@Component({
  template: `
    <p>{{ date | date:'medium' }}</p>
    <p>{{ 1234.56 | number:'1.2-2' }}</p>
  `
})
export class MyComponent {
  date = new Date();
}
```

Angular pipes use `LOCALE_ID` automatically for formatting (dates, numbers, currencies).
