Data binding automatically keeps your page up-to-date based on your application's state. You use data binding to specify things such as the source of an image, the state of a button, or data for a particular user.
Data binding and HTML
Developers can customize HTML by specifying attributes with string values. In the following example, class, src, and disabled modify the <div>, <img>, and <button> elements respectively.

HTML attributes and DOM properties
Angular binding distinguishes between HTML attributes and DOM properties.

Attributes initialize DOM properties and you can configure them to modify an element's behavior. Properties are features of DOM nodes.
A few HTML attributes have 1:1 mapping to properties; for example:
id
Some HTML attributes don't have corresponding properties; for example 
aria-*
Some DOM properties don't have corresponding attributes; for example: 
textContent


Remember that HTML attributes and DOM properties are different things, even when they have the same name.
In Angular, the only role of HTML attributes is to initialize element and directive state.
When you write a data binding, you're dealing exclusively with the DOM properties and events of the target object.

Example 1: an <input>
When the browser renders <input type="text" value="Sarah">, it creates a corresponding DOM node with a value property and initializes that value to "Sarah".
When the user enters Sally into the <input>, the DOM element value property becomes Sally. However, if you look at the HTML attribute value using input.getAttribute('value'), you can see that the attribute remains unchanged —it returns "Sarah".
The HTML attribute value specifies the initial value; the DOM value property is the current value.

Example 2: a disabled button
A button's disabled property is false by default so the button is enabled.
When you add the disabled attribute, you are initializing the button's disabled property to true which disables the button.
Property and attribute comparison
Property: <input [disabled]="condition ? true : false">
Attribute: <input [attr.disabled]="condition ? 'disabled' : null">
Generally, use property binding over attribute binding as a boolean value is easy to read, the syntax is shorter, and a property is more performant.

Types of data binding
Angular provides three categories of data binding according to the direction of data flow:
From source to view ->  {{expression}}  OR  [target]=“expression”
From view to source ->  (target)="statement"
In a two-way sequence of view to source to view -> [(target)]=“expression”

Writable signals
Writable signals provide an API for updating their values directly. You create writable signals by calling the signal function with the signal's initial value:
	const count = signal(0);
To change the value of a writable signal, either .set() it directly:
	count.set(3);
or use the .update() operation to compute a new value from the previous one:
	// Increment the count by 1.
	count.update(value => value + 1);
Writable signals have the type WritableSignal.

Computed signals
Computed signal are read-only signals that derive their value from other signals. You define computed signals using the computed function and specifying a derivation:
	const count: WritableSignal<number> = signal(0);
	const doubleCount: Signal<number> = computed(() => count() * 2);
The doubleCount signal depends on the count signal. 
	If count updates, Angular knows that doubleCount needs to update as well.

Computed signals are both lazily evaluated and memoized
doubleCount's derivation function does not run to calculate its value until the first time you read doubleCount. The calculated value is then cached, and if you read doubleCount again, it will return the cached value without recalculating.
If you then change count, Angular knows that doubleCount's cached value is no longer valid, and the next time you read doubleCount its new value will be calculated.
As a result, you can safely perform computationally expensive derivations in computed signals, such as filtering arrays.

Computed signals are not writable signals
You cannot directly assign values to a computed signal.
	ERROR -> doubleCount.set(3);

Computed signal dependencies are dynamic
Only the signals actually read during the derivation are tracked. 
Reading signals in OnPush components
When you read a signal within an OnPush component's template, Angular tracks the signal as a dependency of that component. When the value of that signal changes, Angular automatically marks the component to ensure it gets updated the next time change detection runs.

Effects
Signals are useful because they notify interested consumers when they change. An effect is an operation that runs whenever one or more signal values change. You can create an effect with the effect function:
	effect(() => {
  		console.log(`The current count is: ${count()}`);
	});
Effects always run at least once. When an effect runs, it tracks any signal value reads. Whenever any of these signal values change, the effect runs again. Similar to computed signals, effects keep track of their dependencies dynamically, and only track signals which were read in the most recent execution.

Effects always execute asynchronously, during the change detection process.

Use cases for effects
Effects are rarely needed in most application code
Logging data being displayed and when it changes, either for analytics or as a debugging tool.
Keeping data in sync with window.localStorage.
Adding custom DOM behavior that can't be expressed with template syntax.
Performing custom rendering to a <canvas>, charting library, or other third party UI library.

When not to use effects
Avoid using effects for propagation of state changes.
Angular by default prevents you from setting signals in effects.
Instead, use computed signals to model state that depends on other state.

Injection context
By default, you can only create an effect() within an injection context
The easiest way to satisfy this requirement is to call effect within a component, directive, or service constructor:
@Component({...})
export class EffectiveCounterComponent {
  readonly count = signal(0);
  constructor() {
    // Register a new effect.
    effect(() => {
      console.log(`The count is: ${this.count()})`);
    });
  }
}

Destroying effects
When you create an effect, it is automatically destroyed when its enclosing context is destroyed.

The syntax for variable declaration, arrays, and classes
3 + 2 + 1 Things
Variable declaration
TypeScript follows the same rules as JavaScript for variable declarations. Variables can be declared using: var, let, and const.

var
Variables in TypeScript can be declared using var keyword, same as in JavaScript. The scoping rules remains the same as in JavaScript.

The variable x was declared within the if block, and yet we were able to access it from outside that block. That’s because var declarations are accessible anywhere within their containing function, module, namespace, or global scope - all which we’ll go over later on - regardless of the containing block. Some people call this var-scoping or function-scoping. Parameters are also function scoped.


let
To solve problems with var declarations, ES6 introduced two new types of variable declarations in JavaScript, using the keywords let and const. TypeScript, being a superset of JavaScript, also supports these new types of variable declarations.
1) Block-scoped let variables cannot be read or written to before they are declared.
2) Let variables cannot be re-declared

Const
Variables can be declared using const similar to var or let declarations. 
The const makes a variable a constant where its value cannot be changed. 

Const variables have the same scoping rules as let variables.
const num:number = 100;
num = 200; //Compiler Error: Cannot assign to 'num' because it is a constant or read-only property

Const variables must be declared and initialized in a single statement. Separate declaration and initialization is not supported.
const num:number; //Compiler Error: const declaration must be initialized
num = 100; 

Const variables allow an object sub-properties to be changed but not the object structure.
Array
There are two ways to declare an array:
1. Using square brackets. This method is similar to how you would declare arrays in JavaScript.
	let fruits: string[] = ['Apple', 'Orange', 'Banana'];
2. Using a generic array type, Array<elementType>.
	let fruits: Array<string> = ['Apple', 'Orange', 'Banana'];

You can initialize an array like below, but you will not get the advantage of TypeScript's type system.
	let arr = [1, 3, 'Apple', 'Orange', 'Banana', true, false];

RxJs - 3 questions
What are Observables and how to use them?
2 Things AAA

What is an Observable?
1 - Observables are objects that can be subscribed to for receiving data. They are a unified way to register callbacks using a subscribe method, which returns an object of type Subscription.
2 - 	fs.readFile(‘<directory>').subscribe(
   		(data) => {// Code that runs once the directory is read},
   		(err) => {  // Code that runs if an error happens}
	);
Observables are often represented as a timeline called a marble diagram. Every marble (circle) represents data being emitted. A straight line represents the completion of the Observable
What is RxJs?
RxJs is a library that aims to standardize how Javascript code handles asynchronous tasks.
Promises were the first attempt in that direction and made their way to the Javascript language.
fs.readFile('<directory>', (err, data) => {
    // Code that runs once the directory is read
});
RxJs is similar to Promises.
How are Observables different from Promises?
Key similarities:
Both use callback functions
Both are used for asynchronous tasks and listening to events that will happen in the future.
Key differences:
A Promise returns only one value. An Observable can (but doesn’t have to) return several values over time.
A Promise starts “working” as soon as it is created. An Observable starts “working” as soon as its subscribe method is called.
We cannot cancel a Promise (it will run no matter what), but we can cancel an Observable.
What is an operator?
Operators are functions that can be used to tweak the behavior of an Observable. There are more than 100 operators in RxJs, and websites such as https://rxmarbles.com/
To use an operator, we call a pipe.
What is a subject?
Subjects are a specific kind of Observable that allows us to send data to several subscribers. In other words, they are multicast. They also offer an easy API to emit data to all subscribers.

let subject = new Subject<string>();
subject.next(“Eureka”); // We use the subject to emit data

With Angular, services typically use subjects to emit new data to all of the components that need that information.
For instance, when we use an @Output in a component, the EventEmitter behind that Output is an RxJs Subject!
Basic operators and usage syntax
1 Thing

1 - RxJS offers numerous built in operators for common use cases such as map(), filter(), concat(), and mergeMap().

const nums = of(1, 2, 3, 4, 5);

// You can create your own custom operator with the pipe method. Create a function that accepts an Observable.
const squareOddVals = pipe(
  filter((n: number) => n % 2 === 1),
  map(n => n * n)
);

// Create an Observable that will run the filter and map functions
const squareOdd = squareOddVals(nums);

// Subscribe to run the combined functions
squareOdd.subscribe(x => console.log(x));

      email: ['', [Validators.required, Validators.email]],
      password: ['', [Validators.required, this.passwordStrengthValidator]],

  // Custom validator for password strength
  passwordStrengthValidator(control: AbstractControl): ValidationErrors | null {
    const value = control.value;
    if (!value) return null;

    const hasUpperCase = /[A-Z]/.test(value);
    const hasLowerCase = /[a-z]/.test(value);
    const hasNumeric = /[0-9]/.test(value);
    const hasSpecialChar = /[!@#$%^&*(),.?":{}|<>]/.test(value);

    const isValid = hasUpperCase && hasLowerCase && hasNumeric && hasSpecialChar && value.length >= 8;
    return isValid ? null : { passwordStrength: true };
  }

  onSubmit() {
    if (this.userForm.valid) {
      console.log('Form submitted:', this.userForm.value);
    }
  }

@Injectable({
  providedIn: 'root'

export class ApiService {
  private apiUrl = 'https://api.example.com';

  constructor(private http: HttpClient) {}

  // GET
  getData<T>(endpoint: string): Observable<T> {
    return this.http.get<T>(`${this.apiUrl}/${endpoint}`);
  }

7. How to Prevent XSS Attacks in Angular
Angular automatically sanitizes values bound to HTML, but you can take extra steps to prevent Cross-Site Scripting (XSS).
A. Angular’s Built-in Sanitization
Angular sanitizes:
{{ }} (interpolation): Escapes HTML by default.
[innerHTML]: Sanitizes HTML content.
[href]/[src]: Validates URLs.
Example: Safe Interpolation
html
Copy
<!-- Safe: Angular escapes HTML -->
<p>{{ userInput }}</p> <!-- Renders as text, not HTML -->

<!-- Unsafe if userInput contains malicious script -->
<p [innerHTML]="userInput"></p> <!-- Angular sanitizes by default -->
B. Bypass Sanitization (Only for Trusted Content)
Use DomSanitizer to mark content as safe only if you trust the source.
import { DomSanitizer, SafeHtml } from '@angular/platform-browser';

@Component({
  selector: 'app-example',
  template: `<div [innerHTML]="safeHtml"></div>`
})
export class ExampleComponent {
  safeHtml: SafeHtml;

  constructor(private sanitizer: DomSanitizer) {
    const maliciousHtml = '<script>alert("XSS Attack!")</script>';
    this.safeHtml = this.sanitizer.bypassSecurityTrustHtml(maliciousHtml); // ⚠️ Only do this if you trust the source!
  }
}
⚠️ Warning: Only use bypassSecurityTrustHtml for fully trusted content (e.g., sanitized by a backend).

C. Disable innerHTML Binding for Untrusted Content
Never bind untrusted user input to [innerHTML].
Use [textContent] or {{ }} instead for plain text.
html

<!-- Safe: Renders as text -->

<!-- Unsafe if userInput is not sanitized -->
<div [innerHTML]="userInput"></div>

CSP is a browser security layer that blocks inline scripts and external resources from untrusted sources.
How to add CSP:

import { Injectable, signal, computed } from '@angular/core';

@Injectable({
  providedIn: 'root', // Singleton service available app-wide

export class CounterService {
  // State: Reactive signal
  private _count = signal(0);

  // Public readonly signal (prevents direct mutation)
  readonly count = this._count.asReadonly();

  // Derived state: Computed from `_count`
  readonly doubleCount = computed(() => this._count() * 2);

  }

🔹 1. Core Concepts
Q1: What is RxJS?
Answer: A library for reactive programming using Observables, enabling asynchronous data streams and event-based logic.
Key Terms:
Observable: Represents a stream of data or events over time (e.g., HTTP responses, user clicks).
Observer: An object with next(), error(), and complete() callbacks to react to Observable emissions.
Subscription: Connects an Observer to an Observable. Must be unsubscribed to avoid memory leaks.
Operators: Pure functions to transform, filter, or combine Observables (e.g., map, filter, mergeMap).

observable.subscribe(console.log); // 'Hello'

🔹 17. How do you implement a custom operator in RxJS?
Use the pipe function to combine existing operators into a reusable custom operator.
Code Example:
typescript
Copy
import { Observable } from 'rxjs';
import { map, filter, pipe } from 'rxjs/operators';

// Custom operator: filter and map in one
function filterAndMap<T, R>(filterFn: (value: T) => boolean, mapFn: (value: T) => R) {
  return pipe(
    filter(filterFn),
    map(mapFn)
  );
}

// Usage
const customOp$ = of(1, 2, 3, 4).pipe(
  filterAndMap(x => x % 2 === 0, x => x * 10)
);
customOp$.subscribe(console.log); // 20, 40

🔹 18. What is the difference between share and publish?
Operator
Behavior
Use Case
share
Multicasts to subscribers and auto-unsubscribes when all subscribers unsubscribe.
Use for hot Observables (e.g., HTTP requests).
publish
Multicasts but does not auto-unsubscribe. Requires refCount: true or manual connect().
Use when you need manual control over the subscription.

Code Example:
typescript
Copy
import { share, publish } from 'rxjs/operators';

const shared$ = someObservable.pipe(share()); // Auto-unsubscribes
const published$ = someObservable.pipe(publish()); // Needs manual connect()
published$.connect(); // Start emitting

🔹 19. How do you implement a retry with exponential backoff?
Use retryWhen with delay and scan to implement exponential backoff for retries.
Code Example:
typescript
Copy
import { retryWhen, delay, scan } from 'rxjs/operators';
import { throwError } from 'rxjs';

this.http.get('/api/data').pipe(
  retryWhen(errors => errors.pipe(
    scan((retryCount, err) => {
      if (retryCount >= 3) {
        throw err; // Max retries reached
      }
      return retryCount + 1;
    }, 0),
    delay(1000 * retryCount) // Exponential backoff: 1s, 2s, 4s, etc.
  ))
).subscribe();

🔹 20. How do you use startWith to provide an initial value for an Observable?
Use startWith to emit an initial value before the Observable starts emitting its own values.
Useful for initializing state in Angular templates.

typescript

import { startWith } from 'rxjs/operators';

const data$ = this.http.get('/api/data').pipe(

data$.subscribe(console.log); // [], [actual data]

1. Patrones Creacionales
Singleton (Instancia Única)
Objetivo: Garantizar que una clase tenga una única instancia y proporcionar un punto de acceso global a ella.
En Angular: Los Servicios con la propiedad providedIn: 'root' implementan este patrón por debajo. Angular se encarga de gestionar que solo exista una instancia para toda la aplicación.
Código en TypeScript / Angular
TypeScript
import { Injectable } from '@angular/core';

@Injectable({
  providedIn: 'root' // <-- Esto convierte al servicio en un Singleton administrado por el DI de Angular
})
export class ConfigService {
  private config: Record<string, string> = {};

  set(key: string, value: string): void {
    this.config[key] = value;
  }

  get(key: string): string {
    return this.config[key];
  }
}

2. Patrones Estructurales
Facade (Fachada)
Objetivo: Proporcionar una interfaz unificada y simplificada para un subsistema complejo (por ejemplo, llamadas a múltiples endpoints de una API o gestión compleja de estados).
En Angular: Muy utilizado para desacoplar los componentes de la lógica de negocio pesada o de los almacenes de estado (NgRx, Akita o Signals). El componente solo habla con la Fachada, y esta se comunica con los servicios de infraestructura.
Código en TypeScript / Angular
TypeScript
import { Injectable, inject, signal } from '@angular/core';
import { HttpClient } from '@angular/common/http';

@Injectable({
  providedIn: 'root'
})
export class UserFacade {
  private http = inject(HttpClient);
  
  // Estado interno expuesto de forma sencilla (usando Angular Signals)
  #usersState = signal<any[]>([]);
  public users = this.#usersState.asReadonly();

  // Simplifica la complejidad de inicializar datos desde múltiples fuentes
  public loadDashboardData(userId: string) {
    this.http.get<any[]>(`/api/users/${userId}/friends`).subscribe(data => {
      this.#usersState.set(data);
    });
    // Podrías lanzar más peticiones en paralelo aquí de forma transparente para el componente
  }
}

3. Patrones de Comportamiento
Observer (Observador)
Objetivo: Definir una suscripción para notificar a múltiples objetos sobre cualquier evento que le suceda al objeto que están observando.
En Angular: Es el núcleo de la programación reactiva del framework. Se encuentra implementado a través de RxJS (Observable, Subject) y, más recientemente, de forma nativa mediante Signals.
Código en TypeScript / Angular (Reactivo)
TypeScript
import { Component, OnInit, inject } from '@angular/core';
import { Subject } from 'rxjs';

// El "Sujeto" (Subject) que emite eventos
@Injectable({ providedIn: 'root' })
export class NotificationService {
  private notificationSource = new Subject<string>();
  
  // El "Observable" al que se suscribirán los "Observadores"
  notifications$ = this.notificationSource.asObservable();

  notify(message: string) {
    this.notificationSource.next(message); // Emite un cambio
  }
}

// El "Observador" (Componente)
@Component({
  selector: 'app-alert',
  template: `<p>Última alerta: {{ currentAlert }}</p>`
})
export class AlertComponent implements OnInit {
  private notifyService = inject(NotificationService);
  currentAlert: string = '';

  ngOnInit() {
    // El componente se registra como observador
    this.notifyService.notifications$.subscribe(msg => {
      this.currentAlert = msg;
    });
  }
}

Strategy (Estrategia)
Objetivo: Define un conjunto de algoritmos intercambiables y permite que el cliente elija cuál usar en tiempo de ejecución de manera dinámica.
En Angular: Ideal para procesar diferentes tipos de renderizado o comportamientos basados en una configuración dinámica (por ejemplo, diferentes métodos de pago o diferentes formatos de exportación de datos).
Código en TypeScript
TypeScript
// 1. Definimos la interfaz común de la estrategia
interface ExportStrategy {
  exportData(data: any[]): void;
}

// 2. Implementamos estrategias concretas
export class CSVExportStrategy implements ExportStrategy {
  exportData(data: any[]) {
    console.log('Exportando a formato CSV...', data);
  }
}

export class JSONExportStrategy implements ExportStrategy {
  exportData(data: any[]) {
    console.log('Exportando a formato JSON...', data);
  }
}

// 3. El Contexto que utiliza la estrategia en Angular
@Injectable({ providedIn: 'root' })
export class DataExporterService {
  private strategy!: ExportStrategy;

  // Permite cambiar la estrategia dinámicamente en tiempo de ejecución
  setStrategy(strategy: ExportStrategy) {
    this.strategy = strategy;
  }

  execute(data: any[]) {
    if (!this.strategy) throw new Error('Estrategia no definida');
    this.strategy.exportData(data);
  }
}

4. Patrones de Arquitectura en Angular
Container / Presentational Components (Smart vs. Dumb)
No forma parte de los patrones clásicos del GoF, pero es un patrón arquitectónico indispensable en aplicaciones SPA modernas como Angular.
Smart Components (Contenedores): Se encargan de la infraestructura. Inyectan servicios, manejan estados, se comunican con APIs externas y pasan datos hacia abajo.
Dumb Components (Presentacionales): No conocen servicios ni APIs. Solo reciben datos mediante @Input() y notifican eventos de interacción del usuario mediante @Output(). Son altamente reutilizables y fáciles de testear.
      [Smart Component] (Maneja datos/servicios)
         /           \
     @Input()      @Output()
       v               ^
 [Dumb Component] [Dumb Component] (Pura UI)

Resumen de arquitectura: Como buena práctica, evita instanciar clases manualmente usando new dentro de Angular para lógica de negocio; deja que el contenedor de Inyección de Dependencias (DI) del framework resuelva las dependencias por ti. Esto respeta el principio de inversión de dependencia (SOLID) y facilita enormemente los tests unitarios.
5. Factory Method (Fábrica)
Objetivo: Creacional. Define una interfaz para crear un objeto, pero deja que las subclases o una lógica condicional decidan qué clase exacta instanciar. Permite delegar la responsabilidad de la instanciación.
En Angular: Es perfecto para cuando necesitas renderizar componentes dinámicos en base a metadatos, o cuando manejas configuraciones dinámicas (como diferentes tipos de drivers de almacenamiento o pasarelas en base al rol de usuario).
Caso de Uso Real: Proveedor dinámico de Logger (Consola vs. Servicio Cloud)
TypeScript
// 1. Interfaz común del producto
export interface Logger {
  log(message: string): void;
}

// 2. Productos concretos
export class ConsoleLogger implements Logger {
  log(message: string) {
    console.log(`[Console]: ${message}`);
  }
}

export class CloudLogger implements Logger {
  log(message: string) {
    // Imagina aquí una petición HTTP real a Datadog, Application Insights, etc.
    console.log(`[Cloud Telemetry]: Enviando a la nube -> ${message}`);
  }
}

// 3. La Fábrica (Factory) integrada con la Inyección de Dependencias de Angular
import { InjectionToken, inject } from '@angular/core';

export const LOGGER_FACTORY = new InjectionToken<Logger>('LoggerFactory', {
  providedIn: 'root',
  factory: () => {
    // Aquí decidimos dinámicamente qué instancia devolver
    // Podríamos usar una variable de entorno, una configuración de usuario, etc.
    const isProduction = true; // Simulación de propiedad de entorno

    return isProduction ? new CloudLogger() : new ConsoleLogger();
  }
});

// 4. Uso en un Componente/Servicio de Angular
@Injectable({ providedIn: 'root' })
export class UserService {
  // Inyectamos el token de la fábrica directamente
  private logger = inject(LOGGER_FACTORY);

  createUser(user: any) {
    // El servicio no sabe (ni le importa) qué tipo de Logger está usando por debajo
    this.logger.log(`Usuario creado: ${user.name}`);
  }
}

6. Adapter (Adaptador)
Objetivo: Estructural. Permite que clases con interfaces incompatibles trabajen juntas. Actúa como un traductor o "envoltorio" entre una API externa (o código heredado) y el formato que tu aplicación espera.
En Angular: Es clave cuando integras librerías de terceros (por ejemplo, de gráficos, mapas o inputs de fecha) o cuando la API de backend cambia el formato del JSON y no quieres romper todos tus componentes front.
Caso de Uso Real: Adaptar una respuesta de API heredada al modelo limpio del Front
TypeScript
// 1. El modelo que mi aplicación Angular prefiere usar (Limpio y tipado)
export interface UserProfile {
  id: string;
  fullName: string;
  emailAddress: string;
}

// 2. La respuesta real de una API antigua/externa (Incompatible o fea)
interface LegacyUserResponse {
  usr_id_v2: number;
  first_name: string;
  last_name: string;
  contact_mail: string;
}

// 3. El Adaptador
@Injectable({ providedIn: 'root' })
export class UserAdapter {
  // Traduce el objeto antiguo al formato nuevo
  adapt(externalUser: LegacyUserResponse): UserProfile {
    return {
      id: externalUser.usr_id_v2.toString(),
      fullName: `${externalUser.first_name} ${externalUser.last_name}`,
      emailAddress: externalUser.contact_mail
    };
  }
}

// 4. Integración en el servicio HTTP de Angular
import { HttpClient } from '@angular/common/http';
import { map, Observable } from 'rxjs';

@Injectable({ providedIn: 'root' })
export class UserApiService {
  private http = inject(HttpClient);
  private adapter = inject(UserAdapter);

  getUser(id: string): Observable<UserProfile> {
    return this.http.get<LegacyUserResponse>(`/api/legacy/users/${id}`).pipe(
      // Aplicamos el adaptador de forma transparente en el flujo reactivo
      map((response) => this.adapter.adapt(response))
    );
  }
}

7. Decorator (Decorador)
Objetivo: Estructural. Añade responsabilidades o comportamientos adicionales a un objeto de forma dinámica sin modificar su código original y sin usar herencia.
En Angular y TS: Los decoradores nativos de TypeScript (@Component, @Injectable, @Directive) son la viva implementación de este patrón a nivel sintáctico (añaden metadatos y lógica de comportamiento a las clases). Pero también podemos construir decoradores personalizados o funcionales para aplicar lógica transversal (AOP - Aspect Oriented Programming) como control de accesos,性能, o manejo de errores.
Caso de Uso Real: Un decorador personalizado para medir el rendimiento de un método
TypeScript
/**
 * Decorador de método que mide cuánto tarda en ejecutarse una función.
 * Nota: En TS moderno (5.x+), la sintaxis estándar de decoradores recibe el método original y un contexto.
 */
export function MeasurePerformance() {
  return function (originalMethod: any, context: ClassMethodDecoratorContext) {
    const methodName = String(context.name);

    function replacementMethod(this: any, ...args: any[]) {
      const start = performance.now();
      
      // Ejecuta el método original con su contexto de clase y argumentos
      const result = originalMethod.apply(this, args);
      
      const end = performance.now();
      console.warn(`[Performance] El método "${methodName}" tardó ${(end - start).toFixed(2)}ms en ejecutarse.`);
      
      return result;
    }

    return replacementMethod;
  };
}

// --- Aplicación en un componente o servicio de Angular ---

@Injectable({ providedIn: 'root' })
export class HeavyDataProcessorService {

  @MeasurePerformance() // <-- Decoramos el método
  processBigArray(items: any[]): any[] {
    // Simulación de un proceso pesado de filtrado o ordenación
    return items
      .filter(item => item.active)
      .map(item => ({ ...item, processedAt: new Date() }));
  }
}

Reflexión de arquitectura:
Usa Factory cuando crear el objeto requiera lógica condicional compleja que no deba conocer el cliente.
Usa Adapter como un muro de contención en tus capas de infraestructura para que los cambios externos (APIs, librerías) no infecten tu lógica de negocio.
Usa Decorator cuando detectes que estás repitiendo la misma lógica (logs, control de errores, loaders) en los métodos de múltiples archivos.
